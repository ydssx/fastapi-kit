# 创作者工作台 MVP 功能补全

## Summary

Creator C 端 MVP 已跑通核心流水线，但项目生命周期、步骤灵活性、发布跟踪与账号自助能力明显偏「demo 级」，影响验证试用时的可信感。本次分两波补齐：**第一波**聚焦工作台内项目/流水线/发布状态（不依赖邮件基础设施）；**第二波**补齐账号设置、登录后改密与邮件找回密码。不改变现有 API 配额规则、流水线定义与自动发帖边界。

---

## Problem Frame

用户反馈与内部试用共识为「功能太简陋」——不是单点 bug，而是缺少创作者会默认期待的基础能力：删错项目、改标题、回退改脚本、看清历史版本、发布 checklist 勾了却看不出哪些平台已发、以及账号侧最基本的改密/找回。当前后端已具备步骤 artifact 多版本与 publish checklist 状态存储，但 API 与 UI 未暴露完整生命周期；密码重置仅 Admin 侧可用。若在验证招募前不补这些缺口，试用者会把产品归类为「概念演示」而非可日常使用的工具。

---

## Actors

- A1. **多平台个人创作者（C 端用户）**：在创作者工作台管理项目、走流水线、核对发布、维护账号。
- A2. **产品/运营（内部）**：验证期可代为重置密码；不面向 C 端暴露 Admin 能力。

---

## Key Flows

- F1. **项目生命周期管理**
  - **Trigger：** 用户创建错了项目、想改标题/平台，或想清理已完成/废弃项目。
  - **Actors：** A1
  - **Steps：** 在项目列表或详情发起编辑/删除 → 系统校验权限与状态规则 → 持久化变更或删除数据。
  - **Outcome：** 用户无需联系支持即可纠正常见误操作。
  - **Covered by：** R1–R3

- F2. **回退与版本管理**
  - **Trigger：** 用户发现前面某步脚本/正文需要修改，或想查看历史确认内容。
  - **Actors：** A1
  - **Steps：** 从步骤条或历史区进入已确认步骤 → 查看完整内容或编辑 → 再次确认时写入新版本 artifact → 后续步骤上下文以最新确认版为准。
  - **Outcome：** 流水线不再是「只能前进、改不了」的线性 demo。
  - **Covered by：** R4–R6

- F3. **发布进度可见**
  - **Trigger：** 用户处于发布核对步骤，已在部分平台完成 checklist。
  - **Actors：** A1
  - **Steps：** 勾选各平台 checklist 项 → 系统按平台聚合完成度 → 列表/详情展示「未发 / 部分已发 / 待发核对」等可读状态。
  - **Outcome：** 符合 [2026-05-22 需求](2026-05-22-creator-ai-workflow-hub-requirements.md) 中「部分已发」的可感知结果，无需自动发帖。
  - **Covered by：** R7–R8

- F4. **账号自助（第二波）**
  - **Trigger：** 用户想改密码，或忘记密码无法登录。
  - **Actors：** A1；A2 仅在邮件不可用或验证期兜底时介入。
  - **Steps（已登录）：** 进入账号设置 → 验证当前密码 → 设置新密码。
  - **Steps（忘记密码）：** 登录页提交邮箱 → 收到一次性重置链接/码 → 设置新密码并登录。
  - **Outcome：** C 端具备基础账号自助，验证期不依赖 Admin 人工重置为常态。
  - **Covered by：** R9–R11

## Requirements

### 第一波：工作台核心（不依赖邮件）

**项目生命周期**
- R1. 用户须能 **删除** 自己的内容项目（进行中或已完成均可），删除前须有一次确认；删除后项目及其步骤 artifact、发布状态、关联埋点引用须从用户视图中移除（硬删除可接受，验证期不要求软删除/回收站）。
- R2. 用户须能 **修改项目标题**；进行中项目随时可改，已完成项目可改标题但不改变只读步骤内容规则（除非 R4 单独允许回退编辑）。
- R3. 用户须能 **修改目标平台列表**，但仅当项目 **尚未进入发布核对步骤** 时允许；若已处于发布步骤或已完成，修改平台须被拒绝并给出明确说明（避免 checklist 与平台不一致）。

**步骤回退与编辑**
- R4. 用户须能从 **步骤进度条或等效导航** 进入任意 **已确认** 的步骤查看完整内容（非仅 200 字预览）。
- R5. 用户对已确认步骤进行编辑并 **再次确认** 时，系统须 **新增 artifact 版本**（保留历史版本），并将该步骤设为当前工作步骤；用户须能继续从该步向后推进（不要求自动作废后续已确认步骤，但后续 AI 上下文须以各步 **最新确认版** 为准）。
- R6. 已完成（`completed`）的项目默认为只读；**不要求** 第一波支持完成后回退编辑（若实现成本低可一并开放，否则明确列为 Out）。

**版本历史**
- R7. 项目详情须提供 **按步骤分组** 的版本历史：每步至少展示版本号、确认时间、完整内容（可折叠/展开或抽屉，须可复制全文）。
- R8. 默认展示每步 **最新确认版**；旧版本须可浏览，**不要求** 第一波支持「一键恢复为当前草稿」按钮（可 deferred）。

**发布跟踪**
- R9. 在 **发布核对步骤**，系统须按 **目标平台** 聚合 checklist 完成情况：某平台下全部项已勾选视为该平台「已发」，部分勾选视为该平台「进行中」，未勾选任何项视为「未发」。
- R10. 项目 **处于发布步骤** 时，列表卡片与详情页须展示可读的 **发布进度摘要**（例如「2 平台中 1 平台已发」或等价文案）；非发布步骤的进行中项目仍显示流水线步骤进度，不显示发布摘要。
- R11. 勾选/取消 checklist 的行为与配额、完成项目的规则 **保持不变**；本次仅增强状态展示，不改变 `complete` 判定逻辑。

### 第二波：账号能力（依赖邮件或验证期兜底）

**账号设置**
- R12. 须提供 **账号设置** 入口（顶栏用户区或等价位置），展示当前登录邮箱（只读即可）。
- R13. 已登录用户须能 **修改密码**，须校验当前密码；成功后须使现有 refresh token 失效或等价安全处理（具体策略留 planning）。

**忘记密码**
- R14. 登录页须提供 **忘记密码** 入口，通过 **邮件一次性链接或验证码** 完成重置；须限流防滥用。
- R15. 若第二波实施时邮件服务尚未就绪，须 documented **验证期兜底**：A2 通过 Admin 重置临时密码（现有能力），C 端登录页展示简短说明；**不得** 静默链向无效流程。

**范围约束（两波通用）**
- R16. 两波交付均 **不得** 改变免费/Pro 配额数值、流水线步骤定义、品牌档案字段、AI 建议契约。
- R17. 两波交付均 **不得** 新增支付、订阅开通、自动发帖或平台 OAuth。

## Acceptance Examples

- **Covers R1.** Given 用户有进行中项目 P，when 在列表对 P 执行删除并确认，then P 不再出现在列表且 GET 详情返回 404。
- **Covers R3.** Given 项目当前在「脚本」步且尚未到发布步，when 用户将目标平台从「小红书」改为「小红书+公众号」，then 保存成功且后续发布 checklist 包含两平台。
- **Covers R3（拒绝路径）.** Given 项目当前步骤为「发布核对」，when 用户尝试修改目标平台，then 操作被拒绝并提示须在更早阶段修改。
- **Covers R4–R5.** Given 用户已确认「钩子/提纲」并进入「口播脚本」，when 用户从步骤条回到「钩子/提纲」编辑并再次确认，then 该步 artifact 版本号递增，且 AI 后续步骤引用的提纲为新版内容。
- **Covers R7.** Given 某步存在 v1、v2 两个 artifact，when 用户打开版本历史，then 可分别查看 v1、v2 全文而非截断预览。
- **Covers R9–R10.** Given 项目目标平台为小红书与公众号，且小红书 checklist 全部勾选、公众号未勾选，when 用户查看项目列表/详情，then 可见「部分已发」或等价发布进度摘要。
- **Covers R13.** Given 用户已登录，when 在账号设置中提交正确当前密码与新密码，then 可用新密码重新登录。
- **Covers R14.** Given 用户忘记密码，when 在登录页提交注册邮箱并点击邮件中的重置链接，then 可设置新密码并成功登录。

## Success Criteria

- **人类结果：** 内部走查 2 条流水线（短视频 + 长图文）时，试用者不再用「太简陋/demo」描述项目删除、改标题、回退改步、看版本、看发布进度；账号侧至少支持登录后改密，忘记密码有邮件流程或 documented 兜底。
- **验证就绪：** 第一波合并后即可启动或继续 [验证期手册](../creator-validation-playbook.md) 招募，不被项目生命周期缺口打断；第二波可在招募初期并行，但须在公测前关闭「只能找 Admin 改密」为常态的情况。
- **下游交付：** `ce-plan` 可按「第一波 / 第二波」拆任务，无需再发明回退语义、部分已发定义或删除策略。

## Scope Boundaries

### Deferred for later

- 项目 **软删除 / 回收站 / 归档库**
- 版本历史 **一键恢复为草稿**、逐步 diff 对比
- **已完成项目** 解锁全量编辑（第一波默认 completed 只读）
- 修改邮箱、双因素认证、第三方登录
- 邮件以外的密码重置渠道（短信等）
- Pro 订阅、支付集成

### Outside this product's identity

- 自动发布、平台 API 对接
- 数据看板、素材库、粉丝互动等 mock 导航能力
- 将 Admin 运维能力嵌入 C 端主界面

## Key Decisions

- **交付节奏：** 方案 B — **第一波** 工作台核心（R1–R11），**第二波** 账号（R12–R15）；同一需求文档，规划时可分两 PR 序列。
- **简陋感根因：** 优先补 **项目/流水线/发布** 可操作性与可见性，而非先做支付或新流水线。
- **删除策略：** 硬删除 + 级联，验证期足够。
- **平台编辑窗口：** 仅发布步骤之前可改目标平台，避免 checklist 状态错乱。
- **回退语义：** 再次确认产生新版本；后续 AI 用最新确认版；不强制清除后续已确认步骤。
- **部分已发：** 由 checklist 按平台聚合推导，不新增「自动发帖」状态机。
- **忘记密码：** 第二波默认 **邮件重置**；SMTP 未就绪时 C 端说明 + Admin 兜底，不做假链接。

## Dependencies / Assumptions

- 依赖现有 Creator MVP（`content_projects`、`project_step_artifacts`、`publish_checklist_state`、JWT 认证）已部署可测。
- 假设第一波 **无需** 新数据库表即可满足 R1–R11（回退步骤可能仅扩展 `current_step_key` 行为与 API，具体留 planning）。
- 假设第二波邮件重置需新增 **SMTP 或事务邮件服务** 配置（仓库当前无邮件能力，已验证）。
- 假设验证期 cohort 规模小，Admin 兜底密码重置可接受为 **临时** 措施。
- 假设与 [原型 Polish 需求](2026-05-25-creator-prototype-fidelity-polish-requirements.md) **并行不冲突** — UI 可复用 polish 后的组件，但本需求以 **行为** 为准。
