Grounding Context

Codebase Context — FastAPI + React Creator 工作台（/creator/），流水线（短视频 / 长图文）+ 品牌档案 + 步骤级 AI 建议 + 终端发布核对清单。小红书以 xiaohongshu key 存在于 target_platforms，新建项目默认勾选；但 AI 生成阶段平台无关，核对规则停留在发布步。

  • 现状：build_step_prompt 未接收 target_platforms（app/services/creator_ai.py:51-61）；XHS checklist 仅 4 项定性描述（app/creator/checklists.py:27-32），对比抖音已有「竖屏 9:16」「标题 ≤30 字」等量化项。
  • 已有杠杆：STEP_AI_ADJUSTMENTS 含「更口语」「更 SEO」等 chip；long_article 流水线有 seo_variants 步；品牌 taboos 已注入 prompt；StepVersionHistory 支持单版本复制。
  • 制度沉淀：平台变更在 publish 步前允许但清空 checklist；发布后锁定平台（40023）；Studio Sidebar + 珊瑚/薄荷色 AI 面板规范。
  • 外部信号（2025-2026）：3:4 封面为流量主规格；标题 9-13 字、关键词前 10 字；标签 5-3-2 策略；CES 权重 保存 > 分享 > 评论；竞品（XHSPlus 等）强调敏感词检测与多平台差异化适配。

#1

Platform Prompt Guard（平台提示词护栏）

Description将 project.target_platforms 传入 build_step_prompt，当含 xiaohongshu 时在 system/user 段注入 XHS 写作约束：口语化 + 分段 + Emoji 适度、首屏钩子、搜索关键词前置。各步骤（script/body/cover/seo）按步骤类型差异化约束，而非一套通用「小红书语气」。

AxisAI 提示词与步骤产出

Basisdirect: creator_ai.py 调用 build_step_prompt 时未传 target_platforms；prompts/__init__.py:18-21 system 模板为平台无关中文助手；前端 adjustment chip 是唯一能「小红书化」的入口。

Rationale这是「定向调整」的最低成本、最高杠杆改动——不改变流水线结构，却让 AI 从第一步起按目标平台产出，消除用户手动点「更口语 / 更 SEO」的补丁式操作。

Downsides多平台项目需定义优先级或分平台变体输出；prompt 变长增加 token；需 A/B 验证生成质量。

Confidence94% High

ComplexityMedium

#2

Checklist-to-Prompt 反哺

Description将 PLATFORM_CHECKLISTS["xiaohongshu"] 的核对项（封面清晰、标题吸引、首屏钩子、话题关键词）在对应前置步骤的 AI prompt 中作为「必须满足的发布标准」注入——例如 body 步要求「首屏 100 字内出现钩子」，cover 步要求「标题含核心搜索词」。发布步 checklist 与生成步 prompt 共用同一规则源（single source of truth）。

AxisAI 提示词与步骤产出 · 发布核对清单与进度

Basisdirect: checklist 定义于 checklists.py:27-32 但仅用于 publish 步勾选；summarize_publish_progress 聚合进度，生成与核对两套标准目前割裂。

Rationale用户不应在发布步才发现「首屏没钩子」——把已有核对语义前移到生成阶段，减少 publish 步返工，且规则变更只需改一处。

Downsides需建立 step_key → checklist_item 映射表；多平台时映射复杂度上升；过度约束可能降低创意多样性。

Confidence88% High

ComplexityMedium

#3

5-3-2 小红书标签生成器

Description扩展 seo_variants 步（或新增 XHS 专用子输出）：当目标含小红书时，AI 返回结构化 JSON——5 个垂类核心词 + 3 个场景细分词 + 2 个时效热搜占位（用户可编辑替换）。UI 以分组 chip 展示，一键复制为 #标签1 #标签2 格式。

AxisAI 提示词与步骤产出

Basisexternal: 2025-2026 小红书标签 5-3-2 策略与 3-5 个标签最优区间；direct: 现有 seo 步 adjustment 为「多平台标题更 SEO」泛化表述（stepAiAdjustments.ts:32-35），无结构化标签输出。

Rationale标签是小红书搜索与推荐的关键杠杆，结构化输出比「一段 SEO 文案」更可操作，且与外部最佳实践对齐。

Downsides热搜词需人工确认或外部数据源；JSON schema 增加前后端契约；多平台时需并列输出而非仅 XHS。

Confidence85% High

ComplexityMedium

#4

量化发布核对（3:4 / 标题 / 首屏）

Description将小红书 checklist 从纯定性升级为可度量项，对齐抖音 checklist 的量化风格：封面「建议 3:4 (1242×1660)」、标题「9-20 字，关键词前 10 字」、正文「首段 ≤100 字含钩子」。发布步 UI 实时显示字数/比例提示（软校验，不阻断保存），勾选前展示达标状态。

Axis发布核对清单与进度 · 格式规范与合规校验

Basisdirect: 抖音已有 title ≤ 30 字、vertical 9:16（checklists.py:21-25），小红书仅「标题吸引点击」等定性项；external: 2026 年 3:4 封面与标题 9-13 字为流量主规格。

Rationale量化核对让用户在勾选前自检，减少「以为达标其实未达」的发布失误；与竞品工具（比例提醒、字数计数）对齐，实现成本低（前端计数 + checklist 文案）。

Downsides软校验可能被忽视；视频笔记 vs 图文笔记规则不同需分支；规格随平台变化需维护常量表。

Confidence91% High

ComplexityLow

#5

一键发布包复制

Description在发布核对步（或项目详情顶栏）增加「复制小红书发布包」：从各步骤 artifact 聚合标题、正文、标签，格式化为 XHS 粘贴友好结构（标题一行、正文分段、标签行）。支持「仅标题+正文」「含标签完整包」两种模式。

Axis创作者工作台前端体验

Basisdirect: StepVersionHistory.tsx 已有单版本复制，但需逐步手动拼接；brainstorm 文档指出「上下文在各工具间复制粘贴」是核心痛点；XHS 发布需跨字段组合。

Rationale工作台价值闭环在「复制到小红书编辑器」——聚合复制是零后端依赖、高感知的功能，直接缩短最后一公里。

Downsidesartifact 字段映射因流水线而异；多平台项目需分平台复制按钮；格式偏好因人而异。

Confidence93% High

#6

敏感词合规扫描层

Description在 body/cover/seo 步编辑器旁增加轻量合规扫描：合并品牌 taboos + 内置小红书高频风险词表（绝对化用语、导流词等），实时高亮并给出替换建议。发布步 checklist 增加「合规扫描已通过」勾选项。

Axis格式规范与合规校验

Basisdirect: 合规目前仅靠品牌 taboos 注入 prompt（BrandPage.tsx 提示绝对化用语）；无自动化扫描；external: XHSPlus 等竞品以敏感词检测为核心卖点。

Rationale小红书对违规词惩罚明确，创作者最怕「写完才发现不能发」——扫描层是差异化功能，可与 taboos 叠加而非替代。

Downsides词库维护与误报；官方规则变化；不能完全替代人工审核；多语言/行业词库扩展成本。

#7

小红书笔记专用流水线

Description新增 xhs_note pipeline（区别于泛用 long_article）：选题 → 爆款标题公式 → 正文（首屏钩子 + 分段）→ 封面视觉要点（3:4 构图）→ 5-3-2 标签 → 发布核对。创建项目时若仅选小红书，默认推荐此流水线；步骤文案与 AI prompt 全部 XHS-native。

Axis平台选择与流水线配置

Basisdirect: 现有仅 short_video / long_article（pipelines.py），新建默认 short_video + xiaohongshu 平台（ProjectsPage.tsx:39-40）——平台与流水线语义错配；reasoned: 小红书图文笔记与短视频脚本、公众号长文的步骤结构本质不同。

Rationale「定向优化」的终极形态是专用流水线——让用户从创建起就走 XHS 最优路径，而非在通用流水线上靠 chip 和 checklist 补丁。

Downsides新 pipeline 维护成本；与多平台并行发布需协调；migration/模板文档工作量大；是 #1-#6 的 superset 而非替代。
